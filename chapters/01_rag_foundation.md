# 第 1 章：RAG 基础与系统边界

## 你要学会什么

RAG 的核心不是“接一个向量数据库”，而是把外部证据接入生成过程，让模型在回答时有可检查、可更新、可权限控制的知识来源。

经典 RAG 论文在 2020 年提出把参数化模型和非参数化外部记忆结合：模型负责语言理解和生成，检索器负责在知识库中找证据。2023-2025 年的综述基本延续这个定义，但把工程栈拆得更细：数据处理、索引、检索、重排、上下文构造、生成、评估、监控和安全。

## 基本流程

```text
raw documents
-> clean and chunk
-> index
-> retrieve(query)
-> rerank
-> build context
-> generate grounded answer
-> evaluate and log trace
```

最小 RAG 只需要三件事：

1. 一个知识库。
2. 一个检索器。
3. 一个只基于证据回答的生成器。

## 为什么不用纯长上下文

长上下文模型降低了“必须检索”的压力，但没有消除 RAG 的价值：

- 私有知识不能全部塞进每次请求。
- 实时知识需要更新，重训模型不现实。
- 权限过滤必须在检索或上下文构造前完成。
- 长上下文存在位置敏感问题，相关证据放在中间时效果可能下降。
- 成本、延迟和审计都要求证据可选择、可记录。

## Demo

运行：

```bash
python demos/01_naive_rag.py
```

这个 demo 做了三件事：

- 用 BM25 从小语料里召回证据。
- 输出 top-k chunk。
- 用模板生成带引用的答案。

你应该观察：

- top-k 是否真的包含答案。
- 答案是否只使用了检索到的证据。
- 引用是否能定位到具体 source 和 chunk id。

## 常见坑

- 只看最终答案，不看检索结果。
- 把“主题相关”误认为“能支持答案”。
- 没有保存 source、版本、权限和 chunk id。
- 资料不足时还强迫模型回答。
